第一章:Web基础  
1.HTTP:
  Web使用HTTP(HyperText Transfer Protocol,超文本传输(转移)协议)的协议作为规范,完成客户端到服务端的运作
2.TCP/IP协议族:
  协议(protocol):约束不同硬件、操作系统之间通信的规则
  TCP/IP协议是与互联网相关的协议集合总称 
    HTTP： HyperText Transfer Protocol 超文本传输协议
    DNS：Domain Name System  负责域名到IP地址之间的解析
    TCP: Transmission Control Protocol 传输控制协议
    UDP: User Data Protocol 用户数据协议
    IP： Internet Protocol 网际协议
  特点:分层
    1.应用层：提供应用服务时通信的活动 (HTTP DNS FTP等就处于改层) 
    2.传输层: 对应用层提供两台计算机之间的数据传输 (TCP和UDP处于该层)
    3.网络层: 处理网络上流动的数据包(网络传输的最小数据单位) 即通过何种路径进行数据传输 (IP处于此层)
    4.数据链路层: 处理硬件部分,如操作系统、网卡、光纤等
    OSI参考模型为七层(表现、会话和物理层)  TCP/IP参考模型为四层
    现在“表现、会话层”属于应用层 “物理层”属于数据链路层
    七层:  由低到高：物数网传会表应 记忆为“物理系有一个人叫数据链路在网络上传输了一个会话来表示应用”
  计算机可被赋予IP地址,也可被赋予主机名和域名 为方便电脑处理和人的记忆,DNS(充当桥梁)可将其互相解析
3.URI:
  URI:Uniform Resourse Indentifier统一资源标识符  标识某一互联网资源
  URL:Uniform Resourse Locator统一资源定位符  标识资源地点,属URI子集
    长度限制:IE(2830) Firefox(65536) Chrome(8182) safari(800000) Opera(19万) 
4.三次握手与四次挥手:
  三次握手: 
    网络传输有延迟,客户端发送的请求若服务端直接返回产生丢包的话会导致客户端因得不到回应超时而关闭,而服务端不知丢包会一直等待客户端的响应,故需三次 主要为了信息传输的可靠性
    client发送SYN请求建立连接  server返回SYNACK同意建立连接  client发送ACK表明收到了同意的回复
  四次挥手:
    client发送FIN=1请求断开连接 server返回ACK=1同意断开连接 server发送FIN=1无资源返回可以断开了 client发送ACK=1表明收到了已无资源返回的回复,可以断开
    四次挥手主要为了防止数据丢失
5.TCP与UDP区别:
  TCP:发送数据前需先建立连接;只支持一对一连接;面向字节流(即数据可拆分多次发送);首部字节大(20字节)
  UDP:发送数据前无需先建立连接;支持一对一,一对多,多对一,多对多;面向报文(即一次发完);首部字节小(8字节)
  应用场景:实时性较高的情况用UDP,如游戏、媒体直播

第二章:HTTP协议
HTTP是无状态协议,即不对请求和响应的通信状态进行保存,即对状态不做持久化处理
  优点:快速处理大量事务,确保协议可伸缩性 (不保存状态就减轻了服务器负担)
  缺点:如在一个网站内请求跳转不同页面,网站无法保存用户状态 (由此引入cookie)  
1.HTTP请求URI资源的方法:
  GET: 获取资源 (也可用来传输实体的主体)
  POST: 传输实体主体
    使用条件:
      请求中包含数据更新等副作用(更新类的页面,如追加、删除等)
      发送敏感信息时(防止Referer安全隐患)
      发送信息量较多时(URL有限制)
  HEAD: 获得报文首部 用于确认URI的有效性及资源更新日期等 (同GET方法一样,只是不返回报文主体部分)
  PUT: 传输文件 (http/1.1版本PUT方法自身不带验证机制,任何人均可上传文件,存在安全问题,一般不用)
  DELETE: 删除文件 (同PUT不带验证机制,一般不用 当配合Web的验证机制或遵守REST标准时可能开放使用)
  OPTIONS: 询问请求的URI资源支持哪些方法
  TRACE: 追踪路径
  CONNECT: 
    要求用隧道协议连接代理(即进行TCP通信) 主要使用SSL(Secure Sockets Layer安全套接层)和
    TLS(Transport Layer Security传输层安全)协议把通信内容加密后经网络隧道传输
  注:方法名区分大小写,需用大写字母

2.持久连接:
  持久连接(HTTP Persistent Connections,也被称为 HTTP keep-alive或HTTP connection reuse)
    旨在建立1次TCP连接后进行多次请求和响应的交互
    特点:只要任意一端未明确提出断开连接,则一直保持TCP连接状态
  管线化: 不用等待响应可直接发送下一个请求,即可同时并行发送多个请求  持久连接使得管线化方式发送请求成为可能

3.HTTP报文:
  用于HTTP协议交互的信息被称为HTTP报文,客户端的报文叫请求报文,响应端的叫响应报文,报文是由多行数据构成的字符串文本  用“CR+LF”作为换行符 (CR:Carriage Return回车符  LF:Line Feed换行符)  
  报文大致分为报文首部和报文主体:
    请求报文结构:请求行(包含用于请求的方法、请求URI和HTTP版本) 请求首部字段 通用首部字段 实体首部字段 其他(可能包含的cookie)
    响应报文结构:状态行(包含表明响应结果的状态码、原因短语和HTTP版本) 响应首部字段 通用首部字段 实体首部字段 其他(可能包含的cookie)
    首部字段:一般有四种,分别为通用、请求、响应和实体首部
    实体:作为请求或响应的有效载荷数据被传输,内容由实体首部和实体主体组成
    报文主体:用于传输请求或响应的实体主体,通常等于实体主体
  编码传输:
    传输过程中通过编码提升传输速率
    常见的内容编码格式: gzip(GUN zip) compress(UNIX系统的标准压缩) deflate(zlib) indentify(不进行编码)
    分块传输编码: 在传输大量数据时通过把数据分割成多块让浏览器逐步显示页面 (即把主体分块)
  发送多种数据的多部分对象集合:
    使用了MIME(Multipurpose Internet Mail Extensions多用途因特网邮件扩展)机制允许邮件处理文本、图片、视频等不同类型的数据  使用方法: Multipart(多部分对象集合)方法
  范围请求:指定范围发送的请求
    为了获取部分内容 使用首部字段Range指定资源byte范围 
    例1: Range: bytes=50-100  例2: Range: bytes=50- (50至全部)  
    例3: Range: bytes=50-100,200-400  例4: bytes=-50 (最后50字节)
    若服务器无法响应范围请求则返回完整的实体内容
  内容协商:
    客户端和服务器端就响应的资源内容进行交涉,然后提供给客户端最合适的资源;以语言、字符集、编码方式为基准判断
    如:浏览器默认语言为英文,当访问既有中文也有英文版本的网站时会优先显示英文版
    协商技术:
      服务器驱动协商(由服务器端进行内容协商)
      客户端驱动协商(由客户端进行内容协商)
      透明协商(服务器端和客户端的结合)

4.HTTP状态码:
  职责:客户端发送请求时描述返回的请求结果,借助状态码了解请求状态
  类别有五种,数量达60多种,经常用的大概14种
  1xx:Informational(信息状态码) 接收的请求正在处理
  2xx:Success(成功状态码) 请求正常处理完毕
    200 OK 请求正常处理
    204 No Content 请求处理成功但无资源返回
    206 Partical Content 客户端进行范围请求,服务器成功处理
  3xx:Redirection(重定向状态码) 需要进行附加操作完成请求
    301 Moved Permanently 永久重定向,资源已被分配新的URI
    302 Found 临时性重定向,资源已被分配新的URI,希望本次请求使用新的URI
    303 See Other 同302,但明确表示客户端应用GET方法获取资源
    304 Not Modified 客户端发送附带条件的请求,服务器找到资源但未符合附加条件
    307 Temporary Redirect 同302
  4xx:Client Error(客户端错误状态码) 服务器无法处理请求
    403 Forbidden 无权限访问
    404 Not Found 服务器无法找到请求的资源 (服务器拒绝且不想给出理由时也可用此状态码)
  5xx:Server Error(服务器错误状态码) 服务器本身处理请求出错   
    500 Internal Server Error 内部资源出故障
    503 Service Unavailable 服务器处于超负载或正在进行停机维护,无法处理请求

5.通信数据转发程序:
  持有资源实体的服务器被称为源服务器
  代理:具有转发功能的应用程序
    接收从客户端发来的请求并转发给服务器,同时也将从服务器返回的响应转发给客户端
    每次通过代理服务器转发请求或响应都会附加“Via”首部字段标记出经过的主机信息
    使用方法有多种,按基准分为两种:是否使用缓存(将资源副本缓存在代理服务器上) 是否会修改报文(不对报文做修改的则称为透明代理,否则非透明)
  网关:转发其他服务器通信数据的服务器
    接收客户端的请求,并像自己拥有源服务器一样处理请求
    能使HTTP请求转化为其他协议通信(即 让服务器提供非HTTP服务)
    可在客户端与网关之间通信线路上加密以确保安全,提高通信安全性
  隧道:为远距离的客户与服务端提供中转,并保持二者的通信连接
    目的确保客户端与服务器端的安全通信 (使用SSL等加密手段)
  缓存:代理服务器或客户端本地磁盘保存的资源副本
    代理转发服务器返回的响应时会保存资源副本,以避免多次转发从源服务器请求的相同资源
    缓存可存在于缓存服务器内,也可客户端缓存(被称为临时文件)


第三章:HTTP首部字段
1.报文结构:报文首部和报文主体(首部和主体之间用空行“CR+LF”隔开)
  请求报文:
    请求行(包含方法、URI及HTTP版本)、请求首部字段、通用首部字段、实体首部字段、其他
    状态行(包含HTTP版本、状态码)、响应首部字段、通用首部字段、实体首部字段、其他

2.首部字段
作用:为浏览器和服务器提供报文主体大小、所使用的语言及认证信息等
  端到端首部:首部会转发给请求/响应对应的最终目标,必须保存在由缓存生成的响应中,它必须被转发
  逐跳首部:首部只对单次转发有效,通过缓存或代理不再转发(如:connection keep-alive等八种)
 1.通用首部字段(General Header Fields):请求和响应都会使用的首部
   1).Cache-Control: 控制缓存的行为
    能否缓存:
      public 其他用户也可利用缓存
      private 对特定用户对象响应
      no-cache 客户端:不接受缓存过的响应 服务端:缓存时需要确认  目的:防止缓存中返回过期资源
      no-store 不进行缓存
	指定缓存期限和认证:
	  max-age=x (单位:秒) 客户端:接收缓存期未超过x秒的资源  服务器端:资源未超过x秒代理无需向源服务器确认
	  min-fresh=x 缓存服务器返回至少在x秒内不得过期的资源
	  s-maxage=x 功能同max-age 只适用于供多位用户使用的公共缓存服务器
	  max-stale=x(x可选) 无参:则无论资源是否过期均接收 有参:若过期未超过x则接收  
	  must-revalidate 代理向源服务器再次验证资源是否有效
	  proxy-revalidate 所有代理接收到请求指令后都要向源服务器再次验证资源是否有效
	  no-transform 无论请求还是响应缓存都不可更改实体主体的媒体类型 (防止缓存或代理压缩图片等类似操作)
   2)Connection: 控制不再转发给代理的首部字段;管理持久连接
	  例:Connection:xxx(不再转发的首部字段名,即代理服务器会将其删掉再转发)  
	  例:Connection:close/Keep-alive(默认)
   3)Date: 表明创建报文的日期和实践
      例:Date: Tue, 03 Jul 2012 04:40:59 GMT (注:时间格式有好几种) 
   4)Pragma:客户端要求所有中间服务器不返回缓存的资源(为遗留字段,效果同Cache-Control)
      例:Cache-Control:no-cache  Pragma:no-cache (可配合使用)
   5)Trailer:事先说明在报文主体后记录了哪些首部字段(可用于分块传输编码)
   6)Transfer-Encoding:规定传输报文主体时采用的编码方式
   7)Upgrade:检测HTTP及其他协议可否使用更高版本来进行通信
	  例:Upgrade: TLS/1.0
   8)Via: 为了追踪客户端和服务端之间经过了哪些代理或网关(经过的代理或网关会在首部添加其自身信息)
   9)Warning: 告知用户一些与缓存相关的问题警告
	  格式: Warning: [警告码] [警告的主机:端口号] [警告内容] ([日期时间])
	  警告码   警告内容							说明
	  110     Response is stale(响应已过期)     代理返回已过期的资源
	  111     Revalidation failed(再验证失败)   代理再验证资源有效性失败
	  112     Disconnection operation(断开连接) 代理与互联网被故意切断
	  113     Heuristic expiration(试探性过期)   响应的使用期大于设定的时间
	  199     Miscellaneous warning(杂项警告)   任意的警告内容
	  214     Transformation applied(使用了转换)  代理对内容编码或媒体类型等进行了某些操作
	  299     Miscellaneous persistent warning(持久杂项警告) 任意警告内容

 2.请求首部字段(Request Header Fields):补充请求的附加内容、客户端信息、响应内容相关优先级等
   1)Accept: 告知服务器用户代理可以处理的媒体类型及优先级
     如:Accept:text/html,image/gif;q=0.8
     注:q代表权重,用“：”分割,范围0-1(最大为1,可精确到小数点3为) 不指定q时默认为1
   2)Accept-Charset:告知服务器用户代理支持的字符集及相对优先顺序
   3)Accept-Encoding:告知服务器用户代理支持的内容编码及优先顺序
	  编码格式:gzip compress deflate identity(不执行压缩或不会变化的默认编码格式)
   4)Accept-Language:告知服务器用户代理可以处理的自然语言集及优先顺序
   5)Authorization:告知服务器用户代理的认证信息(即证书值)
   6)Expect:告知服务器期望出现的某种行为
   7)From:告知服务器期用户的电子邮件地址
   8)Host:告知服务器请求的资源所在的互联网主机名和端口号
   9)If-xxx:条件请求 条件为真时才执行请求
	 如:客户端发送:GET/index.html If-Match:'123'  服务器端有资源index.html,其实体标记(ETag)为“123”  
	 If-Match:当If-Match的字符段与请求资源的ETag一致时才接受请求
	 If-None-Match:作用与If-Match相反
	 If-Modified-Since: 若字段值(即时间)早于资源更新的时间,则处理该请求
	 If-Unmodified-Since: 若字段值(即时间)晚于资源更新的时间,则处理该请求
	 If-Range: 字段值和ETag一致时处理请求 (搭配“Range”字段作为范围请求)
   10)Max-Forwards:服务器每经过一次转发其值减一,当为零时直接返回响应(用于追踪某个代理服务器出现故障)
   11)Proxy-Authorization:代理服务器要客户端认证时,客户端告知认证信息
   12)Range: 获取部分资源的范围请求
   13)Referer: 告知服务器请求资源的原始URI(即请求是从哪个Web页面发送过来的,比如某个网站链接到别的网站的链接)
   14)TE:告知服务器客户端能够处理响应的传输编码方式及优先级(功能与Accept-Encoding相似,但一个对内容编码,一个对传输进行编码)
   15)User-Agent:将发起请求的浏览器及用户代理的信息传达给服务器
 3.响应首部字段(Response Header Fields):补充响应的附加内容、要求客户端附加的额外信息等
 	1)Accept-Range:告知客户端是否能处理范围请求 值:bytes(可以处理)/none(不可处理)
 	2)Age:告知客户端源服务器在多久前创建了响应 (单位:秒)
 	3)ETag:告知客户端实体标识(将资源以字符串形式做标记)
 	  强ETag:实体资源只要有一点变化其值都改变	弱ETag:资源发生根本变化时其值才改变
 	4)Location:将响应接收方引导至与请求URI不一致的位置 (配合3XX提供重定向)
 	5)Proxy-Authenticate:将代理服务器要求的认证信息发送给客户端
 	6)Retry-After:告知客户端在多久以后再次发送请求 (字段值可为秒数或具体日期)
 	7)Server:告知客户端关于服务器的详细信息
 	8)WWW-Authenticate:告知客户端用于访问URI资源的认证方案和带参数提示的质询
 4.实体首部字段(Entity Header Fields):补充资源内容,更新时间等与实体有关的信息
	1)Allow:告知客户端获取请求资源的HTTP方法(如GET、HEAD)
	2)Content-Encoding:告知客户端服务器对实体主体的内容编码方式 (方式有:gzip、compress、deflate、identity)
	3)Content-Language:告知客户端实体主体使用的自然语言
	4)Content-Length:表明实体主体的大小(单位:字节)
	5)Content-Loaction:表明报文主体返回的资源对应的URI
	6)Content-MD5:服务器端对报文主体执行MD5算法后生成的值写入其中,客户端对接收的报文再执行MD5算法后又得到一个值,比较这两个值是否有差异从而判断报文是否完整
	7)Content-Type:表明实体主体内对象的媒体类型
	8)Expiress:告知客户端资源的失效期
	9)Last-Modified:指明资源最终修改的时间
  5.Cookie首部字段:
    1)Set-cookie:服务器开始准备管理客户端状态
    属性:
      NAME=VALUE 赋予Cookie的名称和值
      expiress=DATE Cookie的有效期 (不指明则默认浏览器关闭前有效 注:服务器不可显式删除Cookie但可覆盖)
	  path=PATH 将Cookie保存在服务器上的指定文件目录中
	  domain=域名 将Cookie发送至多个服务器
	  secure 当在HTTPS连接时服务器才回收Cookie记录状其态
	  HttpOnly 限制JS的document.cookie方法读取Cookie内容 (防止XSS攻击)
	2)cookie:status=enable 告知服务器端客户端想要获得状态管理
	3)Chrome查看cookie的插件:Edit this cookie
  6.其他首部字段:
    HTTP首部字段可进行扩展,因此有各种非标字段
    1)X-Frame-Option:防止点击劫持 字段值:DENY(拒绝)/SAMEORIGIN(同源域名下的页面被许可)
	2)X-XSS-Protection:控制浏览器XSS防护机制的开关  0(将XSS过滤设置成无效状态)/1(...有效...)
	3)DNT(Do Not Track):拒绝个人信息被收集(拒绝被精准广告追踪)  0(同意被追踪)/1(拒绝被追踪)


第四章:HTTPS
1.HTTP缺点:
  通信使用明文,内容可被窃听 (通信即使加密,也只是无法破解报文而已,加密后的报文仍可被看到)
    加密手段:通信加密(即使用HTTPS) 报文内容加密
  不验证通信双方身份,可能遭遇伪装
  无法证明报文完整性,报文可能被篡改
    MITM(Man-in-the-Middle atttack中间人攻击):请求或响应在传输途中遭攻击者拦截并篡改内容

2.HTTPS(HTTP Secure):超文本传输安全协议
  HTTPS等于HTTP+加密+认证+完整性保护
  HTTPS并非新协议,而是HTTP通信接口部分用SSL和TLS协议代替而已 (即是身披SSL外壳的HTTP)
    TLS:以SSL为原型开发的协议 有时统一称为SSL
  通过SSL加密,通过第三方机构证书证明身份,发送数据时附加MAC(Message Authentication Code)报文摘要查知报文是否被篡改
  加密方式:
    共享密钥加密(也叫对称密钥加密):加密和解密共用同一个密钥
    两把密钥加密:使用一对非对称的密钥,即一把公钥一把私钥
  HTTPS加密方式:混合加密,即先使用公开密钥交换接下来通信用到的密钥,然后使用共享密钥加密通信
  由自认证机构颁发的证书为自签名证书,非第三方权威机构颁发的证书 (会提示安全警告消息)
  HTTPS加密通信消耗更多CPU及内存,因此在处理速度上比HTTP慢2-100倍(在包含敏感信息时才采用HTTPS)
解析数据包的工具:抓包(Packet Capture)或嗅探器(Sniffer)工具   广泛使用的抓包工具:Wireshark

HTTP默认端口号为80
HTTPS端口号为443

1.确认用户身份:
HTTP1.1认证方式:
  BASIC认证(基本认证):用户ID加密码,对其编码传输  (安全等级低且不灵活,不常用)
  DIGEST认证(摘要认证): (不常用)
  SSL客户端认证:客户端需安装证书
    双因素认证:密码认证+其他信息认证
    SSL采用双因素认证:客户端证书+基于表单验证
  FormBase认证(基于表单认证):即账号+密码
Session与Cookie
  服务器向用户发送Session ID,客户端通过Set-Cookie保存cookie信息  

2.基于HTTP的功能追加协议
  HTTP瓶颈:
    1.一条连接只可发送一个请求
    2.请求只能从客户端开始且客户端不可接收除响应之外的命令
    3.请求/响应首部未经压缩就发送,信息多延迟大
    4.发送冗长首部,每次发送相同首部造成浪费
    5.可任意选择数据压缩格式
  SPDY(取自SPeeDY 发音同speedy)在HTTP功能上进行追加的协议,旨在解决HTTP瓶颈,提升页面加载速度
    运作方式:在TCP/IP的应用层和传输层之间新加会话层,同时以SSL进行通信提高安全
    优点:
      多路复用流:通过单一TCP连接可以无限制处理多个HTTP请求
      赋予请求优先级:可给请求逐个分配优先级顺序
      压缩HTTP请求和响应首部,减少通信流量
      支持服务器主动向客户端推送数据
      服务器可主动提示客户端请求所需的资源
  HTTP2.0:取代了SPDY(实际是基于SPDY改进)
	特点:
	  增强的核心:头信息为二进制分帧传输,更高效紧凑
	  头部信息进行压缩,降低开销
	  并行双向字节流的请求和响应:边请求边响应
	好处:
	  一个连接可并行发送多个请求及响应(即信道复用)
	  并行交错的发送请求,各请求互不影响
	  并行交错的发送响应,各响应互不影响
	  消除不必要延迟,减少加载时间
	  请求分优先级,高优先级先发送
	  服务器主动向客户端发送请求,减少请求延迟
	  默认使用加密
	缺点:队头阻塞、建立连接的握手延迟大
  使用浏览器进行全双工通信的WebSocket新协议:
	一旦客户端与服务器建立WebSocket协议的通信连接,之后所有的通信都依靠这个专用协议进行
	一旦确立此通信连接,客户端和服务器均可直接向对方发送报文
	特点:
	  首部信息小,通信量少
	  只要建立起WebSocket连接,就希望一直保持连接状态
	  服务器可向客户端发送推送功能
	  与之类似的功能:
	    Ajax轮询:客户端每隔一定时间询问服务器是否有新消息,服务器每次予以响应
	    Long Poll轮询:客户端每隔一定时间询问服务器是否有新消息,服务器只在有新消息时响应
  WebDAV(Web-based Distributed Authoring and Versioning基于万维网的分布式创作和版本控制):
	一个可对Web服务器上的内容直接进行文件创建、复制、编辑和删除等操作的分布式文件系统(好比直接修改网盘里的文件)

防火墙:禁止非指定的协议和端口号的数据包通过
CGI(Common Gateway Interface,通用网关接口):Web服务器在收到客户端请求后转发给程序的一组机制
  在CGI的作用下,程序会对内容作出相应的动作,如创建HTNL等动态内容


第六章:Web攻击
1.针对Web应用的攻击模式:主动攻击和被动攻击
  主动攻击:攻击者通过直接访问Web应用,把攻击代码传入的攻击模式,代表性的攻击SQL和OS
  被动攻击:利用圈套策略执行攻击代码的攻击模式,攻击者不直接对Web应用访问发起攻击,代表性攻击XSS和CSRF

2.因输出值转义不完全引发的安全漏洞
  SQL注入攻击:针对Web应用的数据库,通过运行非法的SQL而产生的攻击
    危害:
      非法查看或篡改数据库内的数据
      规避认证
      执行和数据库服务器业务关联的程序等
    防御根本策略:参数化查询(也叫预处理语句)
	  指定查询结构,用户输入预留占位符
	  指定占位符的内容
  OS命令注入攻击(OS Command Injection):通过Web应用,执行非法的操作系统命令
    只要能调用shell函数的地方就存在被攻击的风险
    OS可向Shell发送命令让Windows或Linux操作系统的命令行启动程序
  XSS(Cross-Site Script跨站脚本攻击):
    通过有安全漏洞的Web网站注册用户的浏览器内运行非法的HTML和JS进行的攻击
    危害:
      利用虚假输入表单骗取个人信息
      利用脚本切取Cookie值,帮攻击者发送恶意请求
      显示伪造的文章或图片
    防御策略:
      输入进行验证:输入数据不可过长,且仅包含合法字符,数据可以和一个特殊的正则进行匹配,根据输入的不同字段(如电话、姓名)来限制不同的确认规则
      对输出内容进行编码:将脚本变为内容而非实体进行处理(如空格用字符代替)
  HTTP首部注入攻击:在响应首部插入换行,添加任意响应首部或主体的攻击
    向首部主体添加内容的攻击被称为HTTP响应截断攻击(HTTP Response Splitting Attack)
    危害:
      设置任何Cookie信息
      重定向至任意URL
      显示任意的主体(HTTP响应截断攻击)
  目录遍历攻击:对本无意公开的文件目录,通过非法截断其目录路径后达成访问 (有时也被称为路径遍历攻击)
  远程文件包含漏洞:部分脚本内容需从其他文件读入,利用指定外部服务器的URL充当依赖文件,脚本读取后可运行任意脚本

3.设置或设计上的缺陷引发的安全漏洞
  强制浏览:从Web服务器的公开目录下的文件中浏览那些非自愿公开的文件
    危害:
      泄露顾客的个人信息等
      泄露未连接到外界的文件
      泄露原本需要访问权限的用户才可查阅的信息内容
  不正确的错误消息处理:Web应用抛出的错误提示信息包含对攻击者有用的信息
	Web应用抛出的错误信息
	数据库等系统抛出的错误信息
	  PHP或ASP等脚本的错误
	  数据库或中间件的错误
	  Web服务器的错误
  开放重定向:对指定的任意URL作重定向跳转

4.因会话管理疏忽引发的安全漏洞
  会话劫持:通过某种手段拿到用户的会话ID,并非法使用此会话ID伪装成用户达到攻击
    会话ID记录着客户端的Cookie等信息,服务器端将会话ID与认证状态进行一对一匹配管理
    可能获取会话ID的途径:
      通过非正规的生成方法推测会话ID
      通过窃听或XSS攻击盗取会话ID
      通过会话固定攻击(Session Fixation)强行获取会话ID
  CSRF(Cross-Site Request Forgeries跨站点请求伪造):
    通过陷阱强制对已认证的用户进行个人信息或设定信息等状态更新
    危害:
      利用已通过认证的客户权限更新设定信息等
      利用已通过认证的客户权限购买商品
      利用已通过认证的客户权限在留言板上发表评论
    防御:
      某些操作再增加确认功能
      重新认证,敏感操作让其重新输入密码
      使用Token(详情百度)

5.其他安全漏洞:
  密码破解:
    通过密码网络试错:
      穷举法(暴力破解):将所有的密码组合进行一一验证
      字典攻击:猜测密码在某一组合范围内,对这段范围进行一一验证
    对已加密密码进行破解:攻击者入侵系统,获得加密或散列处理的密码数据的情况
      通过穷举/字典攻击进行类推
      彩虹表:由明文密码及对应的散列值构成的一张数据库表(在彩虹表内搜索散列值找到对应的明文)
      拿到密钥:对于使用共享密钥加密的密码处理数据,拿到其使用的密钥进行解密
      加密算法的漏洞
  Clickjacking(点击劫持):在Web页面上覆盖透明的按钮或链接陷阱,诱导用户点击 (又叫界面伪装UIRedressing)
  DoS攻击(Denial of Service attack):让运行中的服务呈停止状态,也叫服务停止攻击或拒绝服务攻击
    攻击对象不仅限于Web网站,还包括网络设备和服务器
    攻击方式:
      通过攻击安全漏洞使服务停止
      集中利用访问请求造成资源过载,资源用尽的同时实际上服务也就呈停止状态
        即发送大量合法请求,使服务器分辨不出正常请求和攻击请求
    DDoS攻击(Distributed Denial of Service attack):多台计算机发起的DoS攻击
      利用感染病毒的计算机作为攻击跳板
  后门程序:开发设置的隐藏入口,可不按正常步骤使用受限功能
    开发阶段作为Debug调用的后门程序
    开发者为了自身利益植入的后门程序
    攻击者通过某种方法设置的后门程序
